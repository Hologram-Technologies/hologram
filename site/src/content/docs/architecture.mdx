Hologram is structured as a stack of focused layers. Each layer has a single responsibility and depends only on the layer below it.

## System Layers

### LUT & Ring Layer

The foundation. All activation functions and pointwise arithmetic are precomputed into static `[u8; 256]` lookup tables at compile time. Values are encoded as elements of integer rings (Q0, Q1) — no floats on hot paths.

- Activation tables: relu, sigmoid, tanh, gelu, sin, cos, and more
- `ByteRing` — u8 arithmetic with wraparound, closed under add/mul/neg
- Encoding schemes: `AngleEncoding`, `SignedEncoding`, `RawEncoding`
- `ElementWiseView` — applies a LUT to a slice in-place (SIMD-accelerated)
- `no_std + alloc` compatible; compiles for `wasm32-unknown-unknown`

### Expression Graph Layer

Models are expressed as directed acyclic graphs of typed operations. The graph layer handles construction, validation, and optimization before anything runs.

- `HoloGraph` — DAG of `Node` + `Edge` values, each node wrapping a `PrimOp`
- Fusion passes: constant folding, CSE, view fusion
- Scheduling: topological sort → level assignment for parallel execution
- Fluent `GraphBuilder` API for programmatic graph construction

### Compiler Pipeline

Transforms a `HoloGraph` into an executable schedule artifact.

1. Graph validation
2. Constant folding
3. Common subexpression elimination (CSE)
4. View fusion
5. Topological sort + level assignment
6. Liveness analysis (minimizes peak buffer memory)
7. Schedule serialization into a `.holo` archive

### Archive & Loading Layer

Models are stored in `.holo` binary archives — a three-section format (header, graph, weights) using `rkyv` for zero-copy deserialization.

```
┌──────────────────┐
│  Header (fixed)  │  Magic bytes, version, section offsets
├──────────────────┤
│  Graph section   │  Serialized HoloGraph (rkyv)
├──────────────────┤
│  Weight section  │  Quantized weight tensors (rkyv) + SHA-256 checksum
└──────────────────┘
```

Weights are memory-mapped at load time — no copying. `BytesLoader` covers in-memory slices for WASM.

### Execution Layer

Drives inference level by level. Nodes at the same level are independent and run in parallel (Rayon, feature-gated).

```
Level 0: [node_A, node_B]   ← parallel
Level 1: [node_C]           ← depends on A + B
Level 2: [node_D, node_E]   ← parallel
```

Each node dispatch is a KV lookup: `(op_id, encoded_input) → encoded_output`. `BufferArena` provides a bump-allocator for intermediate activations, reset between calls.

**LUT-GEMM**: matrix multiplication is decomposed into row-wise LUT lookups summed via `PsumBook` — eliminating FLOP entirely for quantized weights.

### Async & Streaming

Tokio-based async wrappers around the compiler and executor. The `InferenceStream` type yields partial outputs as a `Stream`, enabling incremental results for large models.

### C & WASM Bindings

- **C ABI** — opaque handles, generated header via `cbindgen`, error propagation via `hologram_last_error()`
- **WASM** — `wasm-bindgen` bindings exposing `HologramSession` as a JavaScript class

## Design Principles

### O(1) Compute

Every activation and pointwise op is a single array index. No branching, no transcendental functions at inference time.

### Ring Algebra

Weights and activations are encoded as integer ring elements. Arithmetic is closed — no float promotion needed.

### KV-Dispatch

Operation ID + encoded operands → precomputed result. The KV registry stores all reachable `(op, input) → output` pairs upfront.

### Zero-Copy Loading

Model weights are mmap'd from `.holo` archives and validated in-place. No deserialization heap allocation.

## Conventions

- Max 3 function arguments; builder pattern for more.
- Functions ≤ 15 lines.
- `macro_rules!` for repeated trait implementations.
- No TODOs, stubs, or `unimplemented!()`.
- Single archive format version — no backwards compatibility shims.
