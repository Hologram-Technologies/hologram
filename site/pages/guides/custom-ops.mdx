# Custom Op Extension API

Added in Sprint 11. Allows registering custom operations that participate in graph construction, fusion, compilation, and KV-dispatch execution.

## Overview

A custom op is defined by implementing the `CustomOp` trait and registering it with the `OpRegistry`. Once registered, it can be referenced in graphs by name and will be compiled + dispatched like any built-in op.

## Implementing a Custom Op

```rust
use hologram::custom_op::{CustomOp, OpContext, OpResult};

struct MySquareOp;

impl CustomOp for MySquareOp {
    fn name(&self) -> &str { "my_square" }

    fn compute(&self, ctx: &OpContext) -> OpResult {
        let x = ctx.input(0)?;
        Ok(x.map(|v| v.wrapping_mul(v)))
    }
}
```

## Registering

```rust
use hologram::registry::OpRegistry;

let mut registry = OpRegistry::global();
registry.register(MySquareOp)?;
```

## Using in a Graph

```rust
let graph = GraphBuilder::new()
    .input("x", Shape::scalar())
    .custom_op("my_square", &["x"], "y")
    .output("y")
    .build()?;
```

## Fusion Hints

Custom ops can opt into view fusion by implementing the optional `fuseable` method:

```rust
fn fuseable(&self) -> bool { true }
```

When `true`, adjacent custom ops with compatible shapes are merged into a single pass by the `view_fusion` optimizer.

## Testing Custom Ops

```rust
#[test]
fn test_my_square() {
    let mut registry = OpRegistry::new();
    registry.register(MySquareOp).unwrap();

    let result = registry.dispatch("my_square", &[Q0::from(3)]).unwrap();
    assert_eq!(result, Q0::from(9));
}
```
