# W8A8 numerics contract

`W8A8_TOKEN_SYM` is the activation treatment the fused output-major decode GEMV
uses. It **changes the computed value**. This document states exactly how, so a
consumer whose V&V is byte-exact against reference transcripts can decide when to
re-baseline, and has something to certify against.

## When it applies

`W8A32` is bit-identical to `dequantize → matmul`, and stays so. `W8A8` is a
different function, so the two take different `op_signature`s — a κ minted under
one can never be served by the other. Switching is a deliberate re-key, visible
in the archive.

**W8A8 is never implicit.** A matmul reaches it exactly when the weight slot sets

```
QuantAttrs::act_quant = W8A8_TOKEN_SYM
```

which defaults to `W8A32`. This is per weight slot — not per graph, per model, or
per release — and nothing in the compiler or the runtime promotes a
declared-`W8A32` weight. It is the same opt-in whether the weight is a graph
constant or bound after compile (a weightless compile).

Witnesses: `wl2_undeclared_const_weight_keeps_w8a32_and_does_not_fuse` and
`w8a8_is_opt_in_undeclared_weights_keep_w8a32_semantics`.

### Two orthogonal declarations

`QuantAttrs` carries two separate things. Conflating them produced a silent wrong
answer, so they are stated apart:

| field | means | who may set it |
|---|---|---|
| `act_quant` | **numerics**: quantize the activation per token (`W8A8_TOKEN_SYM`) or don't (`W8A32`) | any weight |
| `weight_layout` | **layout of the bytes as bound**: `OUTPUT_MAJOR` ⇒ they arrive `[n,k]` | only a weight whose bytes are *not* in the graph |

A constant **with bytes** carries them in `[k,n]`, here, now. `weight_layout =
OUTPUT_MAJOR` on it is a false statement about the graph's own bytes, and the
compiler rejects it (`CompileError::GraphValidation`). To put such a constant on
the fused output-major decode path you set `act_quant` alone — the compiler owns
those bytes and transposes them for you, marking the emitted call `bq_omajor`.

A constant with **no bytes** is a different thing entirely: a
[weightless constant](#weightless-weights) (`ConstantStore::insert_external`)
naming the κ of content that arrives at materialization. Its bytes are not
`[k,n]` — there are none. It is a load-time-bound weight, and declaring
`OUTPUT_MAJOR` for it is the promise that the *binder* will materialize `[n,k]`.
This is the configuration a weight-paging consumer ships, and it is what this
field exists to serve.

### Weightless weights

A weightless compile carries κ, not bytes. `ConstantStore::insert_external(dtype,
shape, kappa)` emits the constant **by reference** with no body, so the archive
holds no weight bytes at all and dedupes across models. `InferenceSession::
load_paged` resolves it through a `WeightProvider`; a fully-resident `load` of a
weightless archive **fails loud** rather than pinning an empty body under the
κ-label of empty content.

The compile-time fusion cannot fire — there is nothing to transpose — so the
*load-time* fusion honours the declaration and emits the same fused
output-major W8A8 call the constant path emits.

Positive end-to-end witness: `tests/weightless_omajor.rs`, which binds the weight
through a provider (not as a graph input, which is a shape nobody ships), runs it
at `m = 1` and `m = 5`, and checks the result against the exact integer oracle
bit for bit.

### The decode gate, and what it means for prefill

For a **constant** weight the compiler only transposes when the matmul is at a
decode shape (`m ≤ decode_gate::OMAJOR_W8A8_MAX_M`); the output-major kernel is a
GEMV and loops rows. Above that gate the constant stays `[k,n]` and the loader
serves it as `W8A32`. So a constant weight declared `W8A8` computes W8A8 at
decode and W8A32 at prefill. That is a value difference bounded by the section
below, and it is deliberate: it buys the prefill a blocked f32 GEMM.

For a **load-time-bound** weight declared `OUTPUT_MAJOR`, the bytes are `[n,k]`
and *no other kernel can read them* — so the output-major kernel runs at every
`m`, decode and prefill alike. It handles `m > 1` by blocking the output
columns, so a block's weight slab is read once and reused by every row rather
than the whole weight being re-streamed per row. Prefill therefore reaches the
same compute ceiling as decode: at a 32 MB weight and `m = 16`, 2.37× the
per-row loop (2.03× at 64 MB); at a cache-resident 4 MB weight the kernel is
compute-bound and the blocking is worth 1.13×. Measure with
`cargo run --release --example i8_m_scaling -p hologram-backend`.

Declaring `OUTPUT_MAJOR` on a weight a prefill stage uses is therefore fine.

The blocking cannot move a κ: every output cell is still one whole dot over the
same `k`-vector, the accumulation is an exact i32 sum, and integer addition is
associative — only the order in which cells are visited changes. Witness:
`batched_integer_gemv_equals_row_by_row_bit_for_bit`, which asserts a batched
call equals `m` independent single-row calls byte for byte, on every lane.

"Runs at every `m`" is executed, not asserted: `prefill_batch_equals_decode_steps_bit_for_bit`
drives a real weightless session at `m = 5` (straddling the `MR = 4` tile) and
requires each prefill row to equal its own decode step bit for bit — and each row
to equal the exact integer oracle. The activation is quantized per token, so this
also pins that one row's `amax` never leaks into another's.

### No silent-wrong

If a weight declares `OUTPUT_MAJOR` and no output-major kernel can serve the call
— unregistered quant tier, `k` past `mm_act_quant::K_MAX`, per-tensor or
group-wise scales, a codebook tier without its codebook, or `act_quant` left at
`W8A32` — it is rejected. Every precondition is static, so the **compiler**
refuses (`CompileError::GraphValidation`, naming the offending predicate); the
loader re-checks the same predicate and refuses too (`ExecError::Backend`), as
defence in depth for archives this compiler did not produce.

Neither falls back. Every other decode path reads `[k,n]`, so a fallback would
transpose the weight by accident and return a plausible, wrong answer.

Witness: `declared_output_major_weight_that_no_kernel_can_serve_fails_loud`.

## What the kernel computes

For output column `j`, with a row-major activation `a[0..k]`, a symmetric
per-channel i8 weight `w[·][j]` and its per-channel scale `sw[j]`:

```
amax    = max |a[i]|                       (per token, per row)
sa      = amax / 127                       (activation scale)
q[i]    = clamp(round_half_away(a[i]/sa), -127, 127)     ∈ i8
ŷ[j]    = (Σ_i q[i] · w[i][j]) · (sa · sw[j])
```

versus the W8A32 reference `y[j] = Σ_i a[i] · (w[i][j] · sw[j])`.

Three facts, in order of importance:

1. **The weight is not re-quantized.** It is already i8; the kernel reads it
   verbatim. No weight error is introduced.
2. **The accumulation is exact.** `Σ q·w` is an i32 sum of products of two i8
   values. The ceiling is *derived* from two declared parameters — the symmetric
   alphabet bound `B = 127` and the accumulator capacity `cap = i32::MAX` — as
   `K_MAX = ⌊cap / B²⌋ = 133_144`. Nothing else in the W8A8 path hard-codes
   `127`; a tier with a different alphabet gets its bound from the same
   expression. The compiler gates on it and the kernel asserts it.

   The bound covers **every intermediate accumulator state**, not just the final
   sum: the reduction materializes each partial sum, and a partial over a prefix
   of length `j ≤ k` is bounded by `j · B² ≤ k · B² ≤ cap`. So no term, no
   partial, and no total can overflow, and integer addition is associative and
   exact. There is no rounding anywhere in the reduction.
3. **The only error is the activation rounding**, plus the single f32 multiply of
   the final writeback.

## The error bound

Activation rounding is the sole source. Round-half-away-from-zero gives

```
|a[i] − q[i]·sa|  ≤  sa/2  =  amax / 254
```

so, writing `Δ[i] = a[i] − q[i]·sa`,

```
|ŷ[j] − y[j]|  =  | Σ_i Δ[i] · w[i][j] · sw[j] |
               ≤  (amax / 254) · sw[j] · Σ_i |w[i][j]|
               ≤  (amax / 254) · sw[j] · 127 · k
               =  (amax · sw[j] · k) / 2
```

The middle form is the useful one: `Σ|w[i][j]|` is known per column and is
normally far below the `127k` worst case. The bound is absolute; relative error
depends on `|y[j]|`, which the substrate cannot bound without knowing the model.

The clamp is not a separate error term: `|a[i]/sa| ≤ 127` by construction, so it
only guards the boundary ulp of the round.

An all-zero activation row is exact: `sa = 0` and the kernel writes zeros.

## Determinism

`ŷ` is **bit-identical across every target** — x86 AVX2, aarch64 NEON, wasm
SIMD128, wasm relaxed-SIMD, and the portable scalar reference. The reduction is an
exact integer sum, so no reassociation can perturb it; the writeback is one
deterministic f32 multiply.

This is now *enforced*, by `integer_gemv_bits_are_the_cross_lane_golden`: the
kernel outputs are hashed and pinned to golden bytes, so whichever lane the runner
compiles must reproduce them. Previously the "conformance sweep" was an
**intra-arch** consistency check (i8 vs i4 vs e8cb, batched vs single-row) inside
one lane, and CI builds this crate for one lane per machine — nothing ever
compared bytes produced on two different lanes. The goldens were captured on
x86-64 AVX2 and verified on aarch64 NEON (qemu) and all three wasm32 tiers.

Consequently W8A8 output is reproducible and κ-stable: the same inputs give the
same bytes and therefore the same content address, on every machine.

### f32 does **not** carry a machine-independent κ

This guarantee is a property of the *integer* path, and does not extend to f32.
`matmul_f32_blocked` fuses its multiply-add on x86 (`_mm256_fmadd_ps`) and NEON
(`vfmaq_f32`); wasm SIMD128 has no fused multiply-add and must compute
`add(mul(..))`, which rounds twice where the fused form rounds once. Measured, on
the same inputs:

```
m=3 k=64 n=24     x86-64 AVX2   aarch64 NEON   wasm32 SIMD128
f32 matmul bits   daefa683…     daefa683…      975651da…   (differs)
i8 / i4 / e8cb    787ccdc1…     787ccdc1…      787ccdc1…   (identical)
```

So an f32 matmul computed in a browser and the same graph computed natively
produce different output bytes, and therefore different content addresses. This is
inherent to f32 + FMA, not a defect to be fixed by reassociating anything: the
only ways out are to forgo FMA everywhere (a real throughput cost) or to accept
that f32 κ is per-lane.

**A consumer whose V&V is byte-exact across native and browser must run the
integer path.** That is the practical reason W8A8 exists, beyond bandwidth.

### Two invariances, and one that does *not* hold

The integer path has two properties the f32 path does not. Both follow from the
accumulation being an exact `i32` sum, and they are worth separating because
they license different optimizations.

**Codec-invariance.** A weight tier is a *codec*: a decode `d : Code → 𝔽` from a
stored alphabet into the working alphabet `{-127..=127}`. The kernel computes
`Σ aᵢ · d(cᵢ)`. So if two codecs decode to the same weight sequence, the result
is the same integer — it is a function of the decoded operands alone, never of
the tier that stored them. i8 is the identity codec; i4 decodes a nibble through
a 16-entry grid; E8CB decodes an index through an 8-D codebook block. **The tier
is a residency/bandwidth choice; the arithmetic identity is fixed.** Witness:
`tiers_that_decode_to_the_same_weights_agree_bit_for_bit`, which feeds all three
kernels one weight matrix and asserts byte-equality, on every lane.

**Schedule-independence.** Integer addition is associative *and* commutative, so
any partition of the operand stream into tiles, in any completion order, reduces
to the same integer. This is why the thread pool's column partition and the wasm
fork-join are bit-identical to serial — and why blocking the output columns to
stream the weight once (see `i8_m_scaling`) cannot move a κ.

**Neither holds for the f32 path.** `matmul_f32_blocked` reduces in `f32`, where
addition is *not* associative and a fused multiply-add rounds once where a
separate multiply-then-add rounds twice. Its summation order is therefore part of
its contract: every output cell must accumulate over `kk` ascending through the
same chain, and an output cell's arithmetic may depend on its column but never on
its row index. Witness: `matmul_row_bytes_are_independent_of_row_index`.

The exactness bound below is what makes the integer invariances bit-level rather
than approximate: under `k ≤ K_MAX` there is no rounding or overflow step at
which two codecs or two schedules could diverge.

## Certifying against it

The oracle is the integer restatement above. It is implemented independently of
the kernels — in `crates/hologram-exec/tests/quantization.rs` and in the kernel
conformance sweeps — and every lane is pinned to it bit-for-bit.

A consumer re-baselining a transcript can:

- assert agreement with the oracle exactly (the substrate does), or
- assert an accuracy metric (cosine, argmax) against the `W8A32` result, using
  the bound above to size the tolerance.

## What is *not* covered

- **Asymmetric weights** (non-zero zero-point) and **group-wise scales**
  (`inner > 1`) do not take this path. They remain fully *correct* through the
  generic dequant path (`dequantize_int8_with_nonzero_zero_point` witnesses it);
  what they lack is the fused integer GEMV, so this is a performance gap, not a
  correctness one. The runtime validates symmetry (all-zero zero-points),
  `inner == 1` and `channels == n` before running the integer GEMV, and either
  falls back or — for a weight that declared `OUTPUT_MAJOR` — fails loud. It
  never silently produces a wrong answer for a weight it cannot decode.
- **E8CB** inherits this contract *on top of* its weight vector-quantization: its
  public entry quantizes the activation internally, so it is **W1A8**, not W1A32.
  A graph using E8CB must therefore set `act_quant = W8A8_TOKEN_SYM`; there is no
  W1A32 form to fall back to, and an E8CB weight left at `W8A32` will not fuse
  and will be rejected at execute time rather than decoded as scalar bytes. Its
  weight error is whatever the model's codebook incurs, which the substrate does
  not bound — only the model's encoder can.
